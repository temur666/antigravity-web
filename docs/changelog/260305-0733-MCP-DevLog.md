## 一、目标

让 Headless LS Daemon（standalone 模式）能加载用户自定义的 MCP Server，使 Agent 在执行任务时可调用用户定义的工具（如 `get_server_status`、`write_note`）。

## 二、已完成的工作

### 2.1 创建了自定义 MCP Server

**文件**: [custom-mcp-server.js](file:///home/tiemuer/antigravity-web/scripts/custom-mcp-server.js)

实现了标准 MCP 协议（JSON-RPC over stdio，Content-Length 分帧），包含两个工具：
- `get_server_status` — 获取服务器运行状态（主机名、内存、磁盘、负载等）
- `write_note` — 在 `tmp/notes/` 目录下创建 Markdown 笔记

手动测试验证（`printf | node custom-mcp-server.js`）**协议握手正常**，`initialize` → 正确返回 capabilities。

### 2.2 配置 MCP Server 注册

发现 LS 读取 MCP 配置的路径是 `~/.gemini/antigravity/mcp_config.json`（**不是** `~/.gemini/settings.json`）。

配置内容：
```json
{
  "mcpServers": {
    "custom-tools": {
      "command": "node",
      "args": ["/home/tiemuer/antigravity-web/scripts/custom-mcp-server.js"]
    }
  }
}
```

## 三、遇到的问题 — MCP 初始化挂死

### 现象

1. LS 成功读取了 `mcp_config.json`，启动了 MCP server 进程（`ps` 可见）
2. `GetMcpServerStates` 返回 `MCP_SERVER_STATUS_PENDING` + `isLoading: true`
3. 等待 60s+ 后状态变为 `MCP_SERVER_STATUS_ERROR`，错误 `context deadline exceeded`
4. MCP server 进程被 LS 杀死

### 分析

| 排查项 | 结果 |
|--------|------|
| MCP server 本身是否正常？ | 正常。手动 `printf ... \| node mcp-server.js` 能正确响应 `initialize` |
| stderr 输出是否干扰？ | 去掉所有 stderr 输出后问题依旧 |
| Content-Length 分帧是否正确？ | 正确，手动测试验证通过 |
| `RefreshMcpServers` 能否重置？ | 调用超时（30s），之后 MCP 变成 `disabled: true` |

### 根因推测

**`standalone=true` 模式可能阻塞了 MCP 子系统的初始化。** 具体来说：

1. LS 在 `standalone` 模式下可能不会执行完整的 MCP 初始化流程
2. MCP 初始化可能依赖某个通过 extension server 订阅推送的前置条件（如 OAuth token、用户设置等）
3. 状态永远停在 PENDING，LS 进程本身不输出任何 MCP 相关日志

**对比**：去掉 `standalone=true` 后，LS 虽然能连接 ext-server 和加载 MCP config，但因为缺少 OAuth token 推送，API 请求被静默丢弃（SendUserCascadeMessage 无响应）。

## 四、替代方案（已验证可行）

### RUN_COMMAND 路径

由于 `autoExecutionPolicy: EAGER` 已全面生效，LS 可以通过 `RUN_COMMAND` 自动执行任意命令。

已创建 CLI 版工具：[cli-tools.js](file:///home/tiemuer/antigravity-web/scripts/cli-tools.js)

```bash
node scripts/cli-tools.js status       # 获取服务器状态
node scripts/cli-tools.js note "标题" "内容"  # 写笔记
```

在 prompt 中提示 LS 使用这些命令即可。
